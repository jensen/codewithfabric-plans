Issue #43: Redirect unverified users to verification page on sign-in instead of showing error

Summary

Currently, when unverified users attempt to sign in, they receive a generic "Invalid email or password" error. This is confusing since their account exists but is not verified. The goal is to detect unverified accounts during sign-in and redirect users to the verification page with appropriate messaging and options.

Root Cause Analysis

Current State

  1. Users sign up with email/password
  2. They receive a verification email but may not complete the process
  3. When they try to sign in, the authorize function in src/lib/auth.ts throws an EmailNotVerified error
  4. The frontend catches this error and displays "Please verify your email before signing in. Check your inbox for the verification link."
  5. However, this is still an error message on the sign-in page, not a helpful redirect

Problems

  • Users see an error message instead of being guided to the solution
  • No direct way to resend verification email from the sign-in page
  • Poor user experience for a common scenario

Proposed Solution

High-Level Approach

  1. Enhanced Detection: Improve the sign-in flow to better handle unverified accounts
  2. Smart Redirection: Redirect unverified users to a dedicated verification page with context
  3. Clear Messaging: Provide explicit guidance about what happened and how to fix it
  4. Resend Option: Include a prominent button to resend the verification email
  5. Seamless Return: After verification, automatically redirect back to sign-in or dashboard

Technical Changes Required

1. Enhanced Sign-In Logic

Modify the sign-in handler to: - Catch the EmailNotVerified error specifically - Store the user's email temporarily (via URL parameter or session storage) - Redirect to a new verification reminder page instead of showing an error

2. New Verification Reminder Page

Create a dedicated page that: - Explains the situation clearly - Shows the email address that needs verification - Provides a resend verification button - Offers troubleshooting options

3. Improved Resend Endpoint

Ensure the resend endpoint can handle the redirect flow properly

4. Post-Verification Handling

After successful verification, redirect to sign-in with a success message

Implementation Steps

Phase 1: Create Unverified Email Page

  1. Create src/app/unverified-email/page.tsx with:
  2. Clear heading: "Email verification required"
  3. Explanation of why they're seeing this page
  4. Display of the email address needing verification
  5. Prominent "Resend verification email" button
  6. Link to contact support if issues persist
  7. Automatic redirect to sign-in after successful verification

Phase 2: Enhance Sign-In Logic

  1. Modify src/app/signin/page.tsx:
  2. In handleEmailSignIn, when catching EmailNotVerified error:
    • Store email in sessionStorage
    • Redirect to /unverified-email?email=${encodeURIComponent(email)}
  3. Remove the inline error display for unverified accounts

Phase 3: Implement Resend Functionality

  1. Update the unverified email page to:
  2. Call the existing /api/auth/resend-verification endpoint
  3. Show success/error feedback
  4. Disable button during request

Phase 4: Post-Verification Flow

  1. Modify src/app/verify-email/page.tsx:
  2. After successful verification, redirect to /signin?message=email-verified
  3. Add success message handling in sign-in page

Phase 5: Testing

  1. Test with unverified account: should redirect to unverified page
  2. Test resend functionality: should show success message
  3. Test verification completion: should redirect to sign-in
  4. Test verified account: should sign in normally
  5. Test edge cases: invalid emails, rate limiting, etc.

Test Strategy

Unit Tests

  • Test error detection in sign-in handler
  • Test redirect logic for unverified users
  • Test email validation in resend endpoint
  • Test sessionStorage usage

Integration Tests

  • Full flow: signup → sign-in attempt → verification → successful sign-in
  • Resend verification email functionality
  • Multiple verification attempts

Edge Cases

  • Invalid email formats
  • Rate limiting on resend requests
  • Expired verification tokens
  • Concurrent verification attempts
  • Mobile vs desktop UX

Risks & Mitigations

Risk Mitigation
Security: Email enumeration through redirect timing Use consistent timing for all sign-in attempts, always redirect after same delay
UX Confusion: Users might not understand why they were redirected Clear messaging explaining the verification requirement and next steps
Rate Limiting: Spam of resend requests Implement server-side rate limiting on resend endpoint
Session Issues: Losing email context during redirects Use both URL parameters and sessionStorage for redundancy
Mobile UX: Poor experience on small screens Design responsive layout, test on various devices

Diagrams

Current Flow vs Proposed Flow

Current Flow:

User signs in → System detects unverified account → Shows error message → User confused

Proposed Flow:

User signs in → System detects unverified account → Redirects to verification reminder page → User understands issue → Can resend email → Completes verification → Successfully signs in

Architecture Diagram

The flow involves: 1. Frontend sign-in form 2. NextAuth authorization 3. Database check for email verification status 4. Redirect to custom verification reminder page 5. Resend API endpoint 6. Email service integration 7. Post-verification redirect back to sign-in

Sequence Diagram

sequenceDiagram
    User->>Frontend: Submit sign-in form
    Frontend->>NextAuth: signIn() call
    NextAuth->>Database: Check user credentials
    Database-->>NextAuth: User found (unverified)
    NextAuth-->>Frontend: EmailNotVerified error
    Frontend->>Frontend: Store email, redirect to /unverified-email
    User->>UnverifiedPage: Views verification reminder
    User->>UnverifiedPage: Click "Resend email"
    UnverifiedPage->>API: POST /resend-verification
    API->>EmailService: Send verification email
    EmailService-->>API: Success
    API-->>UnverifiedPage: Success message
    User->>EmailClient: Open verification link
    EmailClient->>VerifyPage: GET /verify-email?token=...
    VerifyPage->>API: POST /verify-email
    API->>Database: Mark email as verified
    API-->>VerifyPage: Success
    VerifyPage->>SignIn: Redirect with success message

Implementation Timeline

  • Day 1: Create unverified email page and basic redirect logic
  • Day 2: Implement resend functionality and testing
  • Day 3: Add post-verification redirect and final testing
  • Day 4: Edge case testing and deployment

Metrics for Success

  • Reduced support tickets about "invalid credentials" for unverified accounts
  • Increased email verification completion rates
  • Improved user satisfaction scores for sign-in flow
  • Lower bounce rates from sign-in page

Rollback Plan

If issues arise: 1. Feature flag to disable the redirect behavior 2. Fallback to original error message display 3. Monitor error rates and user feedback 4. Hotfix deployment if critical issues found

Future Enhancements

  • Implement magic links for passwordless sign-in
  • Add CAPTCHA to prevent abuse of resend functionality
  • Personalize verification emails based on user behavior
